# Product

## Register

brand

## Platform

web

## Users

Noodle is for developers who need a capable REST client while working close to their code and API files. They use it to create, send, inspect, and iterate on HTTP requests without leaving the terminal or adopting a separate hosted workspace.

## Product Purpose

Noodle is a full-featured terminal REST client. It helps developers keep API requests local, readable, and easy to version-control as YAML files. Success is a developer choosing Noodle as their everyday API workflow because it provides the depth they need without a cloud account, proprietary format, or heavy database.

## Positioning

Noodle makes REST requests portable and version-control friendly by storing them as simple local YAML files rather than locking them into a database, cloud sync service, or proprietary format.

## Conversion & proof

- Primary CTA: Install Noodle.
- Secondary CTAs: Explore the Quick Start and read the documentation.
- The line a visitor remembers after 10 seconds: Noodle is free, open source, and runs entirely in your terminal—without heavy databases, cloud synchronization lock-in, or proprietary formats.
- Belief ladder: Noodle is a capable REST client for everyday API work; YAML files keep requests readable, local, and version-control friendly; environments, imports, history, validation, and automation cover the full workflow; its free, open-source model makes it safe to adopt without lock-in.
- Proof on hand: the project release history in `src/content/changelog/`.

## Brand Personality

Pretty, open source, and full featured. The voice should feel candid and capable: a polished developer tool that is transparent about how it works and confident about the workflow it supports.

## Anti-references

Avoid the feel of commercial REST clients that confine users to cloud-based workspaces, proprietary request formats, or lock-in-heavy account systems.

## Design Principles

1. Make local, readable YAML feel like a practical advantage rather than a compromise.
2. Show feature depth through real developer workflows and product evidence.
3. Keep the experience polished while preserving the directness and transparency associated with open-source tools.
4. Lead visitors toward installation, with documentation and quick-start material close at hand.
5. Let the existing Catppuccin terminal identity remain recognizable across product and documentation surfaces.

## Accessibility & Inclusion

No specific standard was requested. Preserve clear text contrast, keyboard usability, and reduced-motion support as the baseline for future interface work.
